# TS ↔ Rust parity corpus

The `reflect` CLI (`apps/cli`) mirrors a thin read-side contract from
`@reflect/core`: frontmatter splitting and coercions, title derivation, fold
keys, content hashing, and FTS match-expression building. This corpus is the
single source of truth that keeps the two implementations in lockstep.

How it works:

- `notes/` + `daily/` hold small markdown files, one per tricky case (heading
  edge cases, frontmatter coercions, CRLF, unicode, broken YAML, …).
  `scalars.json` holds non-file inputs for `foldKey` and `buildFtsMatch`.
- **`expected.json` is generated by the TypeScript side** —
  `packages/core/src/indexing/parity.test.ts` derives every expectation with
  the real core pipeline (`parseNote` → `buildIndexedNote`, `hashContent`,
  `foldKey`, `buildFtsMatch`) and fails if the committed file differs.
  Regenerate with:

  ```bash
  UPDATE_PARITY=1 pnpm --filter @reflect/core test --run parity
  ```

- **The Rust side only consumes** — `apps/cli/tests/parity.rs` runs the same
  corpus through the CLI's mirror modules and asserts equality with
  `expected.json`.

So a behavior change in core fails the TS test until `expected.json` is
regenerated, and the regenerated file fails the Rust test until the CLI mirror
is updated — both sides must move in the same PR, enforced by the existing CI
jobs. When adding a fixture, add the file (or scalar input), regenerate, and
commit both.

`.gitattributes` disables line-ending conversion: the fixture bytes are the
contract (`crlf.md` deliberately carries `\r\n`).
